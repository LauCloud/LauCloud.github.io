Listener和Filter

发表于 2017-12-24 | 分类于 Java Web | | 阅读次数

这篇文章想要介绍的就是 Listener 和 Filter ，也就是监听器和过滤器了，这两个看起来就是比较高级的技术了，但同时也是非常实用的技术，其实只要从最基本的使用开始入门，还是很好掌握的，掌握之后我们的技能库就又多了两件法宝，碰到相关问题也就又多了一种解决的可能，因此学习新技术还是蛮开心的，下面就开始介绍了，先说 Listener。

1.什么是监听器

其实听名字就可以猜到监听器的作用是什么？那就是监听一个对象，当这个对象有动作的时候，我们监听器也会执行相关动作。当然如果放到 Java 中来说的话，我们所有的事物都是和类相关的，因此我们监听器监听的对象以及我们监听器本身其实都应该是类了，其实在 Java 中监听器就是实现了特定接口的类，可以用来监听另一个类的方法调用或者属性的改变，当被监听对象执行上述的动作时，监听器对象中也会有相应的监听方法执行。

其实知道了监听器是做什么的了之后，就能很好的理解它了，而且对于监听器相关的术语也可以很好的理解，监听器相关的术语有如下几个：

事件源
监听器
事件源和监听器绑定
事件

其实通过一个生活中的小例子，就能很好的理解上面这些术语的意思了，就比如安装在汽车上面的报警器，当有人恶意碰撞我们的车时，报警器就会发出警报声，很显然，报警器是在时刻监视着汽车的，而在这个生活小例子当中，事件源就是汽车，监听器就是我们的报警器了，而事件源和监听器的绑定则是在我们的汽车上面安装报警器了，事件的话就是有人踹了我们的汽车一脚。这样的话，就将生活中的例子和上面的术语一一对应起来了，也能让我们能够更好的理解上面的概念，其实真的理解了这个小例子的话，就会感觉监听器的原理还是蛮简单的。

2.监听器小例子

因为在 Java 的 GUI 图形用户界面编程中监听器还是很常见的，比如一个窗口对象，我们就可以对它进行监听，当它关闭的时候，我们就会执行相应的监听方法了，下面先看代码：

import java.awt.event.WindowEvent;
import java.awt.event.WindowListener;

import javax.swing.JFrame;

public class MyFrame extends JFrame {
    public static void main(String[] args) {
        // 1.创建窗口对象(被监听对象)
        MyFrame myFrame = new MyFrame();
        // 2.设置窗口对象名称
        myFrame.setName("小窗口");
        // 3.设置窗口坐标以及宽高
        myFrame.setBounds(50, 50, 200, 100);
        // 4.设置窗口可见
        myFrame.setVisible(true);
        // 5.绑定监听器
        myFrame.addWindowListener(new MyWindowListener());
    }
}

class MyWindowListener implements WindowListener {

    @Override
    public void windowOpened(WindowEvent e) {
        MyFrame myFrame = (MyFrame) e.getSource();
        System.out.println(myFrame.getName() + "打开啦");
    }

    @Override
    public void windowClosing(WindowEvent e) {
        // TODO Auto-generated method stub
        System.out.println("关闭啦");
    }

    @Override
    public void windowClosed(WindowEvent e) {
        // TODO Auto-generated method stub

    }

    @Override
    public void windowIconified(WindowEvent e) {
        // TODO Auto-generated method stub

    }

    @Override
    public void windowDeiconified(WindowEvent e) {
        // TODO Auto-generated method stub

    }

    @Override
    public void windowActivated(WindowEvent e) {
        // TODO Auto-generated method stub

    }

    @Override
    public void windowDeactivated(WindowEvent e) {
        // TODO Auto-generated method stub

    }
}

首先因为 MyFrame 类继承了 JFrame ，因此由它创建的对象也就是一个窗口对象了，然后就可以调用相关的方法为窗口对象的属性赋值了，下面再看 MyWindowListener 类，实现了 WindowListener 接口，因此也是一个窗口监听器对象了，其中会有很多方法，都是在被监听对象执行某些动作时而触发执行的，比如 windowOpened() 方法，则是在被监听的窗口对象打开时触发的，而 windowClosing() 方法则是在被监听的窗口对象关闭时触发的，监听器对象的方法中都有一个 WindowEvent 对象，这个就是事件对象了，通过事件对象的 getSource() 方法我们就可以拿到被监听对象了，上面的描述中，监听器，被监听对象和事件都被提到了，最后缺的就是被监听对象与监听器的绑定了，上面代码中就是通过被监听对象的 addWindowListener() 方法将被监听对象和监听器进行绑定的。

3.Servlet中的监听器

因为我们做的都是 Java Web 项目，使用 Servlet 的机会会比较多，因此我们就来看看 Servlet 中有哪些监听器呢？其实 Servlet 中的监听器主要是跟 ServletContext、HttpSession 以及 ServletRequest 这三个域对象相关的，也就是说 Servlet 中监听器的事件源主要是上面三个对象。

如果对 Servlet 中的监听器进行分类的话，主要可以分为以下 3 类：

1.监听三个域对象创建和销毁的监听器；
2.监听三个域对象属性变更的监听器，比如属性的增加、删除和替换；
3.监听HttpSession对象中JavaBean状态的改变，比如钝化、活化、绑定以及解除绑定。

下面我们会分别对上述 3 类监听器进行说明。

3.1 关于域对象创建和销毁的监听器

由于是对上面三个域对象进行监听，因此如果想要对这三个域对象创建和销毁进行监听的话，所需要实现的接口也分别是三个，也就是下面这三个了：

ServletContextListener
HttpSessionListener
ServletRequestListener

没错，就是上面这三个接口了，从名字上就可以看出，如果想要监听哪个域对象的创建和销毁的话，应该实现哪个接口了，因此使用起来还是非常简单的，下面直接给出代码分别实现以下对三个域对象创建和销毁的监听。首先是对 ServletContext 域对象的监听。

public class MyServletContextListener implements ServletContextListener {

    @Override
    public void contextDestroyed(ServletContextEvent arg0) {
        System.out.println("ServletContext被销毁了");
    }

    @Override
    public void contextInitialized(ServletContextEvent arg0) {
        System.out.println("ServletContext被初始化了");
    }
}

当我们自己新建的类 MyServletContextListener 实现 ServletContextListener 接口时，那这个类就可以监听到 ServletContext 域对象的创建和销毁了，不过需要使用的话，还需要在 web.xml 文件中进行配置一下，来说明这个类是一个监听器，配置的代码如下：

<listener>
    <listener-class>party.laucloud.weblistener.MyServletContextListener</listener-class>
</listener>

这样就完成对监听器的配置了，而我们知道 ServletContext 域对象是一个项目只有一个的，当项目启动时这个对象会被创建，而项目关闭时该对象会被销毁，在我们实现的监听器中，contextInitialized() 和 contextDestroyed() 方法很显然就是监听该对象创建和销毁的方法了，因此当我们启动和关闭项目时，就可以看到对应的日志打印了。

上面就是 ServletContextListener 接口的使用了，其实还是蛮简单的，只需要实现该接口，然后在 web.xml 进行监听器类的配置就好了，下面我们也将使用 HttpSessionListener 和 ServletRequestListener 来说明它们的作用。

HttpSessionListener

先看相关的代码：

public class MyHttpSessionListener implements HttpSessionListener {

    @Override
    public void sessionCreated(HttpSessionEvent arg0) {
        System.out.println("HttpSession被创建了");
    }

    @Override
    public void sessionDestroyed(HttpSessionEvent arg0) {
        System.out.println("HttpSession被销毁了");
    }
}

和使用 ServletContextListener 接口类似的，先实现 HttpSessionListener 接口，然后新建的类中就会有两个接口中需要重写的方法，那就是 sessionCreated() 和 sessionDestroyed() 方法，很显然这就是监听 HttpSession 对象创建和销毁的方法了，当 HttpSession 对象创建和销毁时我们就可以看到对应方法打印的日志了，当然如果想要使用的话，还必须在 web.xml 配置文件中进行配置。

<listener>
    <listener-class>party.laucloud.weblistener.MyHttpSessionListener</listener-class>
</listener>
<session-config>
    <session-timeout>1</session-timeout>
</session-config>

这样的话我们就可以使用新建的监听器对 HttpSession 对象的创建和销毁进行监听了，下面的配置是设置的 Session 对象被创建之后的过期时间，因为默认是 30 分钟，这里设置为 1 分钟，其实是为了方便后面我们测试，这里需要注意一下的就是 HttpSession 对象什么时候才会创建和销毁呢？

创建：程序在服务器端第一次调用getSession()方法的时候
销毁：
    1.非正常关闭服务器(正常关闭服务器时session对象会被序列化)
    2.session对象的过期时间到了，默认过期时间为30分钟
    3.在程序中调用session.invalidate()方法

还有一个问题就是当我们分别访问 html、JSP 和 Servlet 的时候，这些情况会产生 session 对象吗？其实在访问 html 的时候是不会的，因为没有触发到我们上面所说的 session 对象创建的条件，那访问 JSP 页面的时候呢？其实这个是会的，因为 JSP 是有 session 内置对象的，很显然是会的，那最后的话，访问 Servlet 呢？这个就要看在代码中是否有调用 getSession() 方法了，如果有的话当然就会创建 Session 对象了。说这个的话，主要的目的是为了让我们好测试 MyHttpSessionListener 这个监听器，比如我们可以直接访问一个 JSP 页面，也可以访问一个 Servlet 对象，不过在代码中有创建 Session 和使 Session 失效的代码，这样我们就能看到对应监听器方法所打印的日志了。

ServletRequestListener

下面就是 ServletRequestListener 接口的使用了，首先还是先看代码：

public class MyServletRequestListener implements ServletRequestListener {

    @Override
    public void requestDestroyed(ServletRequestEvent arg0) {
        System.out.println("ServletRequest被销毁了");
    }

    @Override
    public void requestInitialized(ServletRequestEvent arg0) {
        System.out.println("ServletRequest被初始化了");
    }
}

使用上面的话也是和 ServletContextListener 与 HttpSessionListener 类似的，首先还是实现了 ServletRequestListener 接口，然后就需要实现接口中的 requestInitialized() 和 requestDestroyed() 方法，很显然这就是 ServletRequest 被创建和销毁时所要执行的监听方法了，同样的，如果想要使用的话，还需要在 web.xml 文件中进行配置，配置如下：

<listener>
    <listener-class>party.laucloud.weblistener.MyServletRequestListener</listener-class>
</listener>

这样的话我们就可以使用 MyServletRequestListener 这个监听器来对 ServletRequest 对象的创建和销毁进行监听了，那需要注意的就是 ServletRequest 对象在什么情况下会被创建和销毁呢？

创建：当客户端向服务器发送一次请求时，ServletRequest对象就被创建了
销毁：当服务器响应了客户端的请求，那ServletRequest对象就被销毁了

还有一个与 HttpSessionListener 接口中类似的问题，当我们分别访问 html 、 JSP 以及 Servlet 时，ServletRequest 对象会被创建吗？答案是会的，因为访问它们三个时都是一次请求，因此 ServletRequest 对象都会被创建。

3.2 案例：统计当前在线人数

当我们学了上面三个监听域对象创建和销毁的监听器接口时，我们可以用刚学到的知识实现一些功能吗？当然是可以的，比如说统计一个网站的当前在线人数，那这里如何判断一个用户是在线的呢？那就是根据用户的 Session 了，如果一个用户的 Session 是有效的，那我们就认为该用户是在线的，那我们怎么知道有多少个 Session 是有效的呢？那就是前面我们已经使用过的 HttpSessionListener 接口了，实现该接口会有两个监听方法，一个是监听 Session 创建的，而另外一个是监听 Session 销毁的，那我们只需要在 Session 创建时将当前网站在线人数加 1 ，而在 Session 销毁时将当前网站在线人数减 1，这样就可以知道当前网站的在线人数了，其实我们是需要使用一个变量来记录当前网站在线人数的，而且该变量初始化值应该是 0，那我们就可以使用 ServletContextListener 接口了，实现该接口之后重写其中的 contextInitialized() 方法，当应用启动时，在该监听方法中为 ServletContext 域对象添加一个属性，并初始化值为 0,当我们后面使用 HttpSessionListener 接口的实现类时，就可以在它的相关监听方法中对该值进行增减操作了，这也就是实现该案例的原理了。

还是先看代码，首先使用一个 ServletContext 监听器在项目启动时用一个变量来保存当前在线人数，而且默认赋值为 0。

public class OnLineServletContextListener implements ServletContextListener {

    @Override
    public void contextDestroyed(ServletContextEvent event) {

    }

    @Override
    public void contextInitialized(ServletContextEvent event) {
        event.getServletContext().setAttribute("count", 0);
    }
}

当然配置信息还是必不可少的，一样的还是需要在 web.xml 文件中进行配置。

<listener>
    <listener-class>party.laucloud.onlinelistener.OnLineServletContextListener</listener-class>
</listener>

然后的话就是看有多少个用户来访问我们的网站了，因为每一个用户来访问我们网站 JSP 页面时，都会创建一个 Session 对象，因此我们就可以监听 Session 对象的创建和销毁来判断有多少人在线了，具体的监听器代码如下：

public class OnLineHttpSessionListener implements HttpSessionListener {

    @Override
    public void sessionCreated(HttpSessionEvent event) {
        HttpSession session = event.getSession();
        System.out.println(session.getId() + "被创建了");
        int count = (int) session.getServletContext().getAttribute("count");
        count++;
        session.getServletContext().setAttribute("count", count);
    }

    @Override
    public void sessionDestroyed(HttpSessionEvent event) {
        HttpSession session = event.getSession();
        System.out.println(session.getId() + "被销毁了");
        int count = (int) session.getServletContext().getAttribute("count");
        count--;
        session.getServletContext().setAttribute("count", count);
    }
}

同样的，要想使用这个监听器，就还需要在 web.xml 文件中进行配置，而且这里为了测试方便的话，我们可以将 Session 默认的 30 分钟过期时间改小一点，比如只有 1 分钟，这样在测试的时候就能更快的看到效果，配置可以如下：

<listener>
    <listener-class>party.laucloud.onlinelistener.OnLineHttpSessionListener</listener-class>
</listener>
<session-config>
    <session-timeout>1</session-timeout>
</session-config>

而我们需要知道有多少人在线的话，肯定是需要显示出来的，那我们就可以使用一个 JSP 页面来进行显示，其实这里使用 JSP 页面的话还有一个原因，那就是我们在访问 JSP 页面的时候是会创建 Session 对象的，因此这样也能模拟用户登录的效果了，具体的 JSP 页面如下：

<%@ page language="java" contentType="text/html; charset=UTF-8" pageEncoding="UTF-8"%>
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
<html>
<head>
<meta http-equiv="Content-Type" content="text/html; charset=UTF-8">
<title>Insert title here</title>
</head>
<body>
    <h2>当前在线人数：${ count }</h2>
</body>
</html>

记录当前网站在线人数的例子也就写完了，其实最关键的点就是使用 HttpSession 的监听器来监听 Session 对象的创建和销毁。

3.3 关于域对象属性变更的监听器

下面就开始介绍第二类的监听器了，那就是对 ServletContext、HttpSession 以及 ServletRequest 这三个域对象的属性变更进行监听的监听器，所谓的属性变更，其实就是向这三个域对象中添加属性、删除属性以及替换属性，相关的监听器接口就是：

ServletContextAttributeListener
HttpSessionAttributeListener
ServletRequestAttributeListener

看名字的话就可以很明显的看出这三个监听器接口所要监听的域对象是哪个了，下面根据具体的代码演示来分别介绍。

ServletContextAttributeListener

public class MyServletContextAttributeListener implements ServletContextAttributeListener {

    @Override
    public void attributeAdded(ServletContextAttributeEvent arg0) {
        System.out.println("ServletContext对象增加属性了");
    }

    @Override
    public void attributeRemoved(ServletContextAttributeEvent arg0) {
        System.out.println("ServletContext对象移除属性了");
    }

    @Override
    public void attributeReplaced(ServletContextAttributeEvent arg0) {
        System.out.println("ServletContext对象替换属性了");
    }
}

首先是新建一个类实现 ServletContextAttributeListener 接口，然后我们就需要在新建的类中重写接口中的方法，也就是 attributeAdded()、attributeRemoved() 以及 attributeReplaced() 这三个方法，从名字就可以看出是属性增加、属性移除和属性替换的监听方法了，也就是说只要我们在 ServletContext 域对象中增加、移除或者替换属性的话，那我们相应的监听方法也就会执行，我们就可以看到相关的日志打印了。当然，和之前的使用一样，我们还需要在 web.xml 配置文件中进行配置。

<listener>
    <listener-class>party.laucloud.attributelistener.MyServletContextAttributeListener</listener-class>
</listener>

这样的话我们就可以对 ServletContext 属性变更的监听器进行测试了，试着向 ServletContext 中增加、删除或者替换属性，这样我们就能看到监听器中相关方法的日志打印了。

HttpSessionAttributeListener

下面再看 HttpSessionAttributeListener 接口，其实使用的话都是类似的。

public class MyHttpSessionAttributeListener implements HttpSessionAttributeListener {

    @Override
    public void attributeAdded(HttpSessionBindingEvent arg0) {
        System.out.println("HttpSession增加属性");
    }

    @Override
    public void attributeRemoved(HttpSessionBindingEvent arg0) {
        System.out.println("HttpSession移除属性");
    }

    @Override
    public void attributeReplaced(HttpSessionBindingEvent arg0) {
        System.out.println("HttpSession替换属性");
    }
}

同样的，首先先实现 HttpSessionAttributeListener 接口，然后在新建的类中重写接口的方法，同样的 attributeAdded()、attributeRemoved() 和 attributeReplaced() 这三个方法，代表着对 Session 对象中属性的增加、移除以及替换的监听，当 Session 对象对属性有所操作时，我们也就看到相关的日志打印了。同样的，我们还需要在 web.xml 配置文件中进行配置。

<listener>
    <listener-class>party.laucloud.attributelistener.MyHttpSessionAttributeListener</listener-class>
</listener>

这样的话，我们就完成对 HttpSession 对象属性的监听了，因此当我们测试地往 HttpSession 对象中增加、移除或者替换属性时，就可以看到相关日志的打印了。

ServletRequestAttributeListener

下面再看最后一个对 ServletRequest 域对象属性变更进行监听的监听器接口，那就是 ServletRequestAttributeListener，首先还是先看代码。

public class MyServletRequestAttributeListener implements ServletRequestAttributeListener {

    @Override
    public void attributeAdded(ServletRequestAttributeEvent arg0) {
        System.out.println("ServletRequest对象增加属性了");
    }

    @Override
    public void attributeRemoved(ServletRequestAttributeEvent arg0) {
        System.out.println("ServletRequest对象移除属性了");
    }

    @Override
    public void attributeReplaced(ServletRequestAttributeEvent arg0) {
        System.out.println("ServletRequest对象替换属性了");
    }
}

同样的，使用时还是先实现 ServletRequestAttributeListener 接口，然后再重写所实现接口中的方法，也就是 attributeAdded()、attributeRemoved() 以及 attributeReplaced()，这三个方法也就是用来监听 ServletRequest 域对象进行属性增加、属性移除以及属性替换的了，最后也还是需要在 web.xml 配置文件中进行配置。

<listener>
    <listener-class>party.laucloud.attributelistener.MyServletRequestAttributeListener</listener-class>

这样的话我们就能够对实现了 ServletRequestAttributeListener 接口的类进行测试了，通过向 ServletRequest 对象增加、移除或者替换属性，我们就可以看到相关监听方法中的日志打印了。

这样的话，对三个域对象的属性变更进行监听的监听器接口也介绍完了，其实操作起来还是比较简单的，而且三个接口的使用方法非常类似，掌握了一个之后再使用其它的也就十分方便了。

3.4 关于HttpSession对象中JavaBean状态改变的监听器

下面就是第三类的监听器了，这一类的监听器主要关注的就是 JavaBean 在 Session 中的状态了，要想使用这一类监听器，我们首先需要看一下 JavaBean 在 Session 中有哪几种状态？

1.JavaBean对象绑定到Session当中；
2.从Session中解除JavaBean的绑定；
3.JavaBean对象随着Session对象持久化到存储设备当中，比如磁盘，这种状态就叫做钝化；
4.JavaBean对象随着Session对象从存储设备中恢复，这种状态叫活化。

对于上面 JavaBean 的 4 种状态，Servlet 中提供了两个接口让 JavaBean 了解自己在 Session 中的状态，那就是：

HttpSessionBindingListener
HttpSessionActivationListener

从名字就可以看出，第一个接口是与绑定相关的，而第二个接口是与钝化和活化相关的，而且这两个接口与前面我们所介绍的其它监听器接口不同的地方在于，这两个接口在使用的时候是不需要在 web.xml 文件中进行配置的。

我们直接通过代码来查看两个接口的使用，因为这两个接口都是监听 JavaBean 对象的状态的，因此直接使用 JavaBean 实现接口就好了。

HttpSessionBindingListener

public class Student implements HttpSessionBindingListener {
    private String name;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    @Override
    public void valueBound(HttpSessionBindingEvent arg0) {
        System.out.println("Student对象被绑定到HttpSession中了");
    }

    @Override
    public void valueUnbound(HttpSessionBindingEvent arg0) {
        System.out.println("Student对象被HttpSession移除绑定了");
    }
}

首先我们看 Student 类的话，因为实现了 HttpSessionBindingListener，因此需要重写其实现接口中的方法，valueBound() 方法则是监听 JavaBean 对象绑定到 Session 对象中了，而 valueUnbound() 则是监听 JavaBean 对象与 Session 对象解除绑定了，其实我们在测试的时候可以通过查看打印日志，以便了解接口具体的功能是怎样的。

<%
    Student student = new Student();
    session.setAttribute("student", student);
    session.removeAttribute("student");
%>

上面的代码就是实现了 Student 对象与 Session 对象的绑定，并且之后从 Session 中解除了与 Student 对象的绑定，其实上面 JavaBean 与 Session 对象绑定的代码也是为 Session 对象增加属性的代码，但是增加属性的时候可以是任意类型的数据，但是 JavaBean 与 Session 对象绑定则必须是 JavaBean 对象，而不能是像基本数据类型这样的数据了。访问上面的 JSP 页面，从而运行上面的代码，我们就可以在监听器中查看到相应的日志打印了，这样我们就能了解 HttpSessionBindingListener 接口具体实现的功能了。

HttpSessionActivationListener

下面再看 HttpSessionActivationListener 接口的使用，这个接口是关于序列化的，因此我们在新建的 JavaBean 中不仅要实现 HttpSessionActivationListener ，而且还应该实现 Serializable 这个序列化接口。

public class Teacher implements HttpSessionActivationListener, Serializable {
    private String name;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    @Override
    public void sessionDidActivate(HttpSessionEvent arg0) {
        System.out.println("Teacher对象被活化了");
    }

    @Override
    public void sessionWillPassivate(HttpSessionEvent arg0) {
        System.out.println("Teacher对象被钝化了");
    }
}

其实上面已经说到了钝化和活化是怎么一回事了，钝化的话就是我们的 JavaBean 对象随着 Session 对象持久化到存储设备中，而活化的话则相反，就是 JavaBean 对象随着 Session 对象从存储设备中恢复过来，那这里就有一个问题了，什么时候 Session 对象会持久化到存储设备中呢？其实当我们正常关闭服务器的时候就会，但是我们为了测试方便，也可以使用配置的方式，就是配置 Session 对象多长时间不被访问的话就会持久化到存储设备当中，其实就是钝化了，而我们再次访问 Session 对象时，持久化的 Session 对象就会从存储设备中恢复过来，这时就是活化了，具体的配置就是在我们工程的 WebContent 目录下的 META-INF 新建一个 context.xml 配置文件，文件中的内容如下：

<?xml version="1.0" encoding="UTF-8"?>
<Context>
    <Manager className="org.apache.catalina.session.PersistentManager"
        maxIdleSwap="1">
        <Store className="org.apache.catalina.session.FileStore"
            directory="laucloud" />
    </Manager>
</Context>

其实上面的配置就是关于 Session 对象持久化的配置，maxIdleSwap 这个参数就是设置的 Session 对象在多长时间不被访问时就会被持久化到存储设备中，这里是设置的 1 分钟，还有 directory 就是我们持久化 Session 保存的目录，因为我们的工程名是 ListenerFilter，因此我们就可以在 Tomcat 服务器安装目录的 apache-tomcat-7.0.57\work\Catalina\localhost\ListenerFilter 这个地址中，看到有一个 laucloud 目录，这里就是存放的持久化 Session。因此我们现在的设置就是 Session 对象在创建之后 1 分钟不被访问的话，就会持久化到存储设备中。

因此就可以在 JSP 页面中先运行如下的代码：

<%
    Teacher teacher = new Teacher();
    teacher.setName("Kobe");
    session.setAttribute("teacher", teacher);
%>

这里就是将一个 Teacher 对象绑定到 Session 对象上面了，如果就这样等 1 分钟的话，Session 对象就会持久化，而我们的 Teacher 对象也会随着一起持久化，那这时候我们就可以看到监听 JavaBean 对象随着 Session 对象一起持久化的日志打印了，如果我们这时候再到另一个 JSP 中进行如下访问的话：

<%@ page language="java" contentType="text/html; charset=UTF-8"
pageEncoding="UTF-8"%>
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
<html>
<head>
<meta http-equiv="Content-Type" content="text/html; charset=UTF-8">
<title>Insert title here</title>
</head>
<body>
    ${ teacher.name }
</body>
</html>

这时候 JavaBean 对象就会随着 Session 对象从持久化设备中恢复过来，也就是活化了，那我们也可以看到监听 JavaBean 对象活化的日志打印了。

这里还需要说明的一点就是关于上面 Session 序列化配置的，其实不仅可以在工程的 META-INF 目录进行配置，其实还有两个地方也能进行配置：

1.tomcat/conf/context.xml
2.tomcat/conf/Catalina/localhost/context.xml

第 1 个地方就是直接在 Tomcat 目录下面的 conf 目录进行配置了，在这里进行配置的话，就是 Tomcat 下面所有的虚拟主机和虚拟目录下面的工程都会序列化 Session；而第 2 种配置是在 localhost 主机下面进行的配置，因此 localhost 虚拟主机下面的所有工程都会序列化 Session；我们在上面使用的，在工程里面配置的方式则是只针对当前项目的。这里有一点需要注意的就是为什么要配置 Session 对象自动序列化到存储设备当中呢？其实这是因为怕我们网站中的 Session 对象过多，占用大量的内存资源，而又不是每个 Session 对象都会立即使用，因此可以在过一段时间之后将 Session 对象序列化到存储设备当中，这样的话就不会占用大量的内存资源，而且当我们再想使用已经序列号的 Session 对象时，也是可以直接从存储设备中恢复过来的，因此还是可以解决大量 Session 对象存在但是又不马上使用的问题的。

4.Filter过滤器

上面已经介绍完了 Listener 监听器，那我们下面就开始介绍 Servlet 中另一个十分实用的技术，那就是过滤器。那什么是过滤器呢？其实就是用来过滤客户端到服务器端的请求的，那为什么要过滤呢？举个简单的例子就很明了了，比如说用户还没登录时，却在浏览器地址栏输入登录成功的页面地址，这时候如果不对登陆成功页面的请求进行过滤的话，那我们网站的安全性可能就会有威胁，因为可能登录成功页面是可以对数据进行修改的，因此还是应该使用一个过滤器来检查用户是否已经登录了。

4.1 过滤器入门

首先通过一个最简单的例子来学习过滤器最基本的使用：

public class FilterDemo implements Filter {

    @Override
    public void destroy() {

    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {
        System.out.println("FilterDemo执行了");
        chain.doFilter(request, response);
        System.out.println("FilterDemo执行结束了");
    }

    @Override
    public void init(FilterConfig config) throws ServletException {

    }
}

要想得到一个过滤器类就必须实现 javax.servlet.Filter 接口，这样的话就表明该类是一个过滤器了，实现了接口之后就必须重写接口当中的方法，也就是 init()、doFilter() 以及 destroy() 这三个方法了，根据我们之前学习域对象的知识，很容易看出其实这是关于过滤器生命周期的方法，也就是初始化、进行过滤以及最后进行销毁的方法。当然我们只需要关注进行过滤的 doFilter() 方法就好了，会看到上面的方法中有关于日志打印的，然后就是 chain.doFilter(request, response); 进行请求放行的代码了，其实这句代码的意思就是当前请求是允许通过的。当然和使用监听器类似的，要想使用过滤器的话，也需要在 web.xml 配置文件中进行配置，配置如下：

<filter>
    <filter-name>FilterDemo</filter-name>
    <filter-class>party.laucloud.filter.demo.FilterDemo</filter-class>
</filter>
<filter-mapping>
    <filter-name>FilterDemo</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

上面的 filter-class 是配置的哪个类是过滤器类，而下面的 url-pattern 则是表示我们这个过滤器会对哪些请求进行过滤，/* 当然就是表示所有的请求都会被过滤了，而是否允许通过其实还是在代码中进行判断的，也就是由 FilterChain 来进行放行操作的。

4.2 FilterChain对象

4.3 Filter的生命周期

关于 Filter 的生命周期的话，可以看我们上面实现 Filter 接口之后重写的方法，init() 则是表示初始化的，doFilter() 则是表示过滤操作，是否允许请求通行则是在该方法里执行，而 destroy() 方法则是表示销毁的。其实 Filter 对象的创建和销毁是由 Web 服务器来负责的，服务器启动时，就会创建 Filter 实例对象，并且调用 init() 方法进行初始化操作，每次进行请求过滤时，则是调用的 doFilter() 方法，在这个方法中判断是否允许请求通过，最后服务器关闭时，Filter 对象就会被销毁，而 destroy() 方法就会被执行。

4.4 FilterConfig对象

和 ServletConfig 对象类似的，FilterConfig 对象也是用来获取配置信息的，当然是用来获取 Filter 的配置信息了，当我们在 web.xml 文件中配置好 Filter 之后，就可以使用 Filter 过滤器对象了，这时候我们可以看到在 init() 初始化方法中是有 FilterConfig 类型的参数的，我们也是在这里来获取 Filter 的相关配置的，先给出在 web.xml 中的配置：

<filter>
    <filter-name>FilterConfigDemo</filter-name>
    <filter-class>party.laucloud.filter.demo.FilterConfigDemo</filter-class>
    <init-param>
        <param-name>username</param-name>
        <param-value>root</param-value>
    </init-param>
    <init-param>
        <param-name>password</param-name>
        <param-value>123456</param-value>
    </init-param>
</filter>
<filter-mapping>
    <filter-name>FilterConfigDemo</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

然后再到 init() 方法中进行参数的获取，具体的代码如下：

public class FilterConfigDemo implements Filter {

    @Override
    public void destroy() {

    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {

    }

    @Override
    public void init(FilterConfig filterConfig) throws ServletException {
        String filterName = filterConfig.getFilterName();
        System.out.println("name:" + filterName);
        String username = filterConfig.getInitParameter("username");
        String password = filterConfig.getInitParameter("password");
        System.out.println(username + "--" + password);

        Enumeration<String> names = filterConfig.getInitParameterNames();
        while (names.hasMoreElements()) {
            String name = (String) names.nextElement();
            String value = filterConfig.getInitParameter(name);
            System.out.println(name + "--" + value);
        }
    }
}

这样的话我们就可以获取得到在 web.xml 配置文件中配置的参数信息了。其实如果还想了解更多关于 FilterConfig 对象的信息的话，可以查询一下对应的 api 文档，其实除了我们上面使用的 3 个方法之外，就还有一个 getServletContext() 方法，这个的话我们就非常熟悉了，其实就是用来获取 ServletContext 对象的。

4.5 Filter的相关配置信息

下面我们就来看看关于 Filter 对象相关的配置信息，了解这些配置信息的规则，可以让我们使用 Filter 时更加得心应手。

url-pattern的配置

url-pattern 的配置分为 3 类，如下所示：

完全路径匹配：以/开始，例如/aaa,/aaa/bbb
目录匹配：以/开始，而以*结尾，例如/*,/aaa/*
扩展名匹配：不是以/开始，而是以*开始，例如*.jsp，*.do，*.action

完全路径匹配表示的是过滤请求路径完全相同的请求，而目录匹配则是表示匹配在某个目录下的所有请求，最后的扩展名匹配则是过滤以某些后缀结尾的请求。

servlet-name的配置

在 filter-mapping 的配置当中，其实我们还可以直接使用 servlet-name 的配置，这样配置的话就是表示这个 Filter 过滤的就是对于该 Servlet 的请求。

dispatcher的配置

在之前的配置 Filter 的时候，我们在 filter-mapping 里面是没有看到 dispatcher 的配置的，但是这个配置还是起了作用的，就是取的默认值 REQUEST，也就是相当于如下的配置：

<dispatcher>REQUEST</dispatcher>

那这个配置是什么意思呢？其实就是说我们当前这个 Filter 过滤器会对匹配 url-pattern 中设置的路径的请求进行过滤，但是这里需要注意的是，因为配置的是 REQUEST，因此过滤的也只是请求，而别的就不会过滤了，比如请求转发、页面包含和出现错误时进行的页面跳转。其实上面的说的也就是 dispatcher 这个配置可以取的枚举值了。

REQUEST：默认值。也就是普通的请求；
FORWARD：请求转发时进行过滤；
INCLUDE：页面包含时进行过滤；
ERROR：页面出现全局错误时页面跳转时进行过滤。

5.Filter案例

下面通过两个小的案例来巩固我们学过的 Filter 过滤器知识。

5.1 过滤对成功页面的请求

这里先说明一下具体的需求，比如说有 2 个 JSP 页面，一个是登录页面 login.jsp ，在 WebContent 目录下，第二个是登录成功页面 success.jsp 页面，在 WebContent 下的 jsp 目录下面，如果不进行设置的话，其实我们是可以直接访问到登录成功页面的，那如果只能在登录页面登录成功之后才能访问呢？其实是可以在 success.jsp 中加一段代码的，来判断 Session 对象中是否含有用户信息，但是如果页面比较多的话，那这个工作量就太大了，而且代码也十分不友好，因此我们可以考虑使用 Filter 来完成这个需求。

具体的 Filter 代码如下：

public class LoginFilter implements Filter {

    @Override
    public void destroy() {

    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        HttpSession session = req.getSession();
        if (null != session.getAttribute("existUser")) {
            chain.doFilter(req, response);
        } else {
            req.getRequestDispatcher("/login.jsp").forward(req, response);
        }
    }

    @Override
    public void init(FilterConfig arg0) throws ServletException {

    }
}

相关配置如下：

<filter>
    <filter-name>LoginFilter</filter-name>
    <filter-class>party.laucloud.filter.LoginFilter</filter-class>
</filter>
<filter-mapping>
    <filter-name>LoginFilter</filter-name>
    <url-pattern>/jsp/*</url-pattern>
</filter-mapping>

这样的话就可以完成我们所说的功能了，Filter 对于 jsp 目录下面的请求都会进行过滤，因此就可以在 Filter 中进行判断，看 Session 中是否已经有用户信息，有的话则直接放行，没有的话的返回到登录页面进行登录操作。

5.2 通用的中文字符乱码处理器

我们知道当我们在 Servlet 中接受中文字符时，接受到的往往都是乱码，而且如果要处理的话，对于 GET 请求方式和 POST 请求方式还是不一样的，而我们想做的就是不管是 GET 请求方式，还是 POST 请求方式，都能够接受到正常的中文字符。

其实对于上面的需求来说，我们之前的客户端向服务器端发送的都是 HttpServletRequest，我们使用这个对象来接受中文字符时是有问题的，因此就需要增强原来的 HttpServletRequest 类，让增强的类具有接受中文字符而不乱码的能力，这里就需要说一下增强一个类的几种方式了。

继承：需要控制被增强类的构造器
装饰者模式：
    被增强的类和增强的类需要实现相同的接口
    在增强的类中能够获取到被增强类的引用
    缺点：接口中方法过多，需要重写很多方法
动态代理：类需要实现接口

这里的话我们就使用装饰者模式来增强原来的 HttpServletRequest 类了，我们可以新建一个类 MyHttpServletRequest ，在这个类里面肯定是可以拿到 HttpServletRequest 类的引用的，比如将 HttpServletRequest 类的引用设置为一个成员变量，然后设计一个构造器是需要传入原来的 HttpServletRequest 类引用的，关于装饰者模式的缺点的话，确实 HttpServletRequest 中的接口方法还是非常多的，如果我们的 MyHttpServletRequest 直接重写所有的方法的话，其实大部分都还是使用的 HttpServletRequest 中原来的功能，而我们其实只会改写 getParameter() 这个方法，不过非常幸运的是，HttpServletRequest 有一个实现类，也是一个包装类，里面的方法都已经重写了，那就是 HttpServletRequestWrapper 类，所以我们就可以直接让 MyHttpServletRequest 类继承于这个包装类，然后修改其中的 getParameter() 方法，这样的话我们就能拿到想要的增强类了，而又不必重写大量的接口方法。

MyHttpServletRequest

public class MyHttpServletRequest extends HttpServletRequestWrapper {

    private HttpServletRequest request;

    public MyHttpServletRequest(HttpServletRequest request) {
        super(request);
        this.request = request;
    }

    @Override
    public String getParameter(String name) {
        String method = request.getMethod();
        if ("get".equalsIgnoreCase(method)) {
            String origName = super.getParameter(name);
            String parseName = "";
            try {
                parseName = new String(origName.getBytes("ISO-8859-1"), "UTF-8");
            } catch (UnsupportedEncodingException e) {
                e.printStackTrace();
            }
            return parseName;
        } else if ("post".equalsIgnoreCase(method)) {
            try {
                request.setCharacterEncoding("UTF-8");
            } catch (UnsupportedEncodingException e) {
                e.printStackTrace();
            }
        }
        return super.getParameter(name);
    }
}

再看一下我们的 Filter 中的代码：

GenericEncodingFilter

public class GenericEncodingFilter implements Filter {

    @Override
    public void destroy() {

    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        MyHttpServletRequest req = new MyHttpServletRequest(httpRequest);
        chain.doFilter(req, response);
    }

    @Override
    public void init(FilterConfig arg0) throws ServletException {

    }
}

相关 Filter 的配置的话，当然是过滤所有的请求了，因此在 web.xml 文件中的配置应该是：

<filter>
    <filter-name>GenericEncodingFilter</filter-name>
    <filter-class>party.laucloud.filter.GenericEncodingFilter</filter-class>
</filter>
<filter-mapping>
    <filter-name>GenericEncodingFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

这样的话，就完成了一个通用的接受中文字符乱码的过滤器了，当我们在 Servlet 中接受中文字符时就不会再出现乱码了，因此我们的需求就可以得到解决了。

6.总结

其实像 Listener 和 filter 这样的技术，在开发中还是非常实用的，不管是统计网站当前在线人数，还是接受中文字符乱码统一的过滤器，都可以看出其用处还是蛮大的，而且在很多框架之中，这两个技术也会运用的到，在之后学习框架的过程中，我们能掌握一些这样比较底层的技术，对于我们理解框架的使用也会很有帮助，而且也可以更好地理解别人的设计为什么是好的，这样就相当于在为自己以后的学习打基础了，因此还是非常重要的。
